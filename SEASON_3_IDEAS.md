# StockGame Season 3 실행 아이디어 (개정안)

## 0) 시즌 3 핵심 원칙
- **시장 범위 단순화**: 시즌 3는 **국내 증시(KOSPI/KOSDAQ)만** 지원한다.
- **포지션 단순화**: **롱(BUY/SELL)만** 지원하고 공매도/숏 포지션은 완전 제거한다.
- **신용/미수는 규칙화**: `신용한도`, `미수잔액`, `정산 예정건(T+2 영업일)`을 분리해 추적한다.
- **일관성 우선**: 프론트/백엔드/스케줄러/미션/리더보드의 용어와 계산식이 동일해야 한다.

## 1) 공매도 완전 제거 (Short 제거)

### 1-1. 제거 대상 (코드 레벨 체크리스트)
- **거래 엔진**
  - `data_engine/trade_executor.py`: `SHORT`, `COVER`, 음수 수량(`quantity < 0`) 관련 분기 제거.
  - `frontend/lib/trade.ts`: 동일하게 `SHORT/COVER` 분기 제거.
- **UI**
  - `frontend/components/TradeModal.tsx`: 숏 안내 문구/숏 계산/숏 관련 버튼 라벨 제거.
  - `frontend/components/PortfolioTable.tsx`: `SHORT` 배지 제거.
  - `frontend/components/Leaderboard.tsx`: `positionMode=short` 토글 제거, 숏 집계 제거.
  - `frontend/components/DashboardOverview.tsx`, `frontend/components/AssetAllocationChart.tsx`: shortValue/shortInitialValue 전제 계산 제거.
- **미션**
  - `data_engine/mission_manager.py`: `bear_market_bet`, `short_cover_profit` 제거.
- **정산/스케줄러**
  - `data_engine/scheduler_rtdb.py`: 숏 커버 자동청산 fallback 로직 제거.

### 1-2. 데이터 규칙
- 포트폴리오 수량은 **항상 0 이상**.
- 거래 타입은 `BUY`, `SELL`, `REWARD`(+ 필요 시 `LIMIT_EXECUTED`)로 축소.
- 시즌 3 시작 시점에는 기존 시즌 데이터 리셋(`data_engine/reset_season.py`)을 기본 전제로 한다.

### 2-1. 제거 대상 (코드 레벨 체크리스트)
- **수집/시세**
  - `data_engine/scheduler_rtdb.py`: `fetch_us_stocks`, `is_us_market_open`, USD 환율 기반 매매 계산 제거.

  - `frontend/components/TradeModal.tsx`, `frontend/lib/trade.ts`, `data_engine/trade_executor.py`의 USD/KRW 환산 분기 제거.
  - `data_engine/scheduler_rtdb.py`의 지정가 주문 `currency` 분기(USD/KRW 비교) 제거.
- **UI**
  - `frontend/components/Ticker.tsx`: 시즌 3에서는 국내 지수 중심으로 단순화(USD/KRW 표시 옵션화).

### 2-2. 데이터 규칙
- `market`은 `KOSPI`, `KOSDAQ`(또는 기존 `KRX` 통일값)만 허용.
- 종목 심볼은 국내 규칙(숫자형 코드)만 허용.
- 거래/랭킹/포트폴리오 계산은 전부 KRW 단일 통화로 처리.

## 3) 신용한도/미수/T+2(2영업일) 강화

### 3-1. 정책 명확화
- 모든 유저 `creditLimit = 0` 으로 시즌 시작.
- 신용한도 획득 경로:
  - 일일 미션 보상
  - 미니게임 보상
- **정산 기준은 “48시간”이 아니라 “T+2 영업일”**로 고정한다.
  - 영업일 계산은 한국 거래일 기준.
  - 주말/공휴일은 만기일에서 제외.

### 3-2. 권장 데이터 모델
- `users/{uid}`
  - `creditLimit` (총 한도)
  - `usedCredit` (현재 사용 중인 신용)
  - `unsettledAmount` (정산 대기 미수 총액)
- `users/{uid}/credit_ledger/{ledgerId}` (신규 권장)
  - `tradeId`, `symbol`, `principal`, `openedAt`, `dueDate`, `status(PENDING|SETTLED|LIQUIDATED)`, `settledAt`

### 3-3. 정산 절차 (스케줄러)
- 일 배치에서 `dueDate <= today` AND `status=PENDING` 건을 처리.
- 처리 순서:
  1. 예수금으로 우선 상환
  2. 부족분은 자동 반대매매로 상환
  3. 상환 후 ledger 상태 갱신
- 자동 반대매매 정책(고정 규칙 필요):
  - 권장: **최근 매수분 우선(LIFO)** 또는 **평가금액 큰 순** 중 하나로 고정.
- 상환 실패(유동성 부족) 시:
  - 신규 매수 차단
  - 운영자 점검 플래그 기록(계정 상태 표시)

### 3-4. 이자/수수료 정합성
- 화면 표시 수수료와 실제 계산 수수료를 동일하게 맞춘다.
- 이자 부과 주기/기준(일 단위)을 문서화하고 UI에 동일 문구로 노출한다.

## 4) 즉시 진행 기능 (시즌 3 본편)

### 4-1. 시즌 종료 D-Day
- `RTDB system/season_end` 기준으로 상단 카운트다운 표시.

### 4-2. 리더보드 강화
- 전체 보유 비중 시각화는 **롱 포지션 전용**으로 단순화.
- 대형 거래 뉴스 티커:
  - 예: 체결 금액 1억 이상 `BUY/SELL` 이벤트를 실시간 노출.
  - 개인정보 노출 레벨(닉네임/익명) 정책 별도 정의.
- Treemap 자산맵 추가: 사용자들의 자산 크기를 영역 비율로 표시하는 비주얼 추가.


### 4-3. 미니게임 (캔들 예측 퀴즈)
- 시즌 3에서는 “신용한도 성장 장치” 역할로 연결.
- 연승 보상과 쿨다운 정책을 먼저 고정한 뒤, UI/애니메이션 구현.
- 실제 과거 주가 데이터(캔들 차트) 10~20개를 무작위로 보여줌.
- 유저는 바로 다음에 올 캔들이 **'양봉(상승)'**일지 **'음봉(하락)'**일지 맞춤.
- 연속 성공 횟수에 따라 신용한도 영구 상향 (예: 3연승 시 +1,000만 원).

## 5) 보류 항목 (시즌 3.5+)

### 5-1. AI 참가자(자동 매매 봇)
- **보류 이유**: 전략 품질보다 운영 리스크(비용, 반복성, 이상 주문 제어)가 큼.
- **어려운 지점**
  - LLM 응답 변동성으로 인한 매매 일관성 저하
  - 비정상 주문 방지(포지션/한도/리스크 가드레일) 설계 부담
  - 실시간 호출 비용 및 장애 대응 복잡도
- **기술 구조**: 스케줄러를 통해 일정 시간마다 시장 데이터와 봇의 페르소나를 LLM에 전달하여 매매 결정(Loop). 
  - **다양한 페르소나**: 
    - **워런 버핏 봇**: 저평가 우량주 장기 보유.
    - **불나방 봇**: 급등주 위주 공격적 단타.
    - **안전제일 봇**: 지수 추종 및 대형주 위주.

### 5-2. 자산 위탁/카피트레이딩
- **보류 이유**: 정산/수수료/해지/동기화까지 포함하면 거래 코어 난이도가 급격히 상승.
- **어려운 지점**
  - 매니저-위탁자 비율 동기화 시 체결 오차 처리
  - 수익배분 정산 기준 시점(실현/평가) 정의
  - 해지 시 대량 청산으로 인한 시장왜곡/체결 충돌
- 카피트레이딩: 위탁자가 자산의 일부(예: 10~50%)를 특정 매니저에게 맡기면, 매니저의 매매 비중과 동일하게 위탁자의 계좌도 자동 매매됨.
- 성과 보수: 수익이 발생한 경우에만 수익금의 일부(예: 10%)를 매니저에게 포인트나 신용한도 상향 등으로 지급.
- 위탁 해지: 위탁자는 언제든 해지 가능하나, 해지 시점에 매니저가 보유 중인 종목은 즉시 현재가 매도 정산 처리.

## 7) UI/UX 프리미엄 디자인 및 대시보드 강화
- **디자인 컨셉**: 
    - "Dark Mode + Glassmorphism"을 기반으로 한 고유하고 프리미엄한 디자인.
    - 복잡한 지표를 덜어내고, PPT 장표처럼 핵심 데이터가 한눈에 들어오는 **카드형 대시보드** 레이아웃.
- **자산 구성 시각화 (Simplified Finance View)**:
    - **핵심 3대 지표**: `순자산(Equity)`, `예수금(Cash)`, `주식 평가금(Stocks)`을 가장 크게 강조.
    - **부채 가시화**: `미수금(Debt)`과 `신용한도(Credit Limit)`를 별도의 '위험도 게이지'와 함께 표시하여 계산 과정을 명확히 전달.
    - **보유 비중 차트**: 기존의 단순 리스트를 넘어, 섹터별/통화별(KRW vs USD) 비중을 간결한 차트로 요약하여 '포트폴리오 요약 카드' 제공.
- **PPT 스타일 마이페이지**:
    - 스크롤을 내리며 하나씩 정보를 읽는 방식이 아닌, 한 화면(Single Page)에 정리된 보고서 형태로 모든 정보를 배치.
    - 애니메이션 효과(Counting Up, Chart Drawing)를 적극 활용하여 '살아있는 데이터' 느낌 부여.
    - 인포그래픽 활용
- **차트 시스템 현대화 (On-demand API 방식)**:
    - **Supabase 의존성 제거**: 역사적 주가 데이터를 Supabase에 전수 저장하는 방식에서, 필요할 때 실시간으로 네이버 API를 호출하는 방식으로 전환. (DB 비용 및 관리 리스크 최소화)
    - **데이터 수급 전략**: 네이버 API의 페이징(pageSize, page) 기능을 활용하여 최근 60일(1페이지)부터 과거 데이터까지 필요한 만큼 동적으로 병합.
    - **프리미엄 커스텀 UI**: 네이버 화면을 복사하는 것이 아니라, 순수 데이터를 받아 **Recharts/ApexCharts** 등으로 게임 톤앤매너에 맞는 고유 차트 구현. (PPT 느낌의 세련된 디자인)
    - **인터랙션 강화**: 지연 로딩(Lazy Loading)을 통해 사용자가 과거로 스크롤할 때만 추가 데이터를 불러오는 스마트한 UX 제공. 현재 호가/체결 정보와 연동하여 실시간성 부여.

## 8) 거래 안정성/공정성 리스크 점검 (시즌3 필수)

### 8-1. 핵심 거래 로직 이원화(프론트/백엔드 중복) 리스크
- **현 상태**
  - 거래 핵심 로직이 `frontend/lib/trade.ts`와 `data_engine/trade_executor.py`에 동시에 존재한다.
  - 같은 정책을 두 군데에서 유지해야 해서, 수정 시 쉽게 드리프트가 발생한다.
- **실제 위험**
  - 시즌3에서 공매도/해외주식 제거를 한쪽만 반영하면 주문 결과가 경로별로 달라질 수 있다.
  - 지정가 주문은 스케줄러 경유(백엔드 로직), 시장가는 클라이언트 경유(프론트 로직)라서 불일치가 더 잘 발생한다.
- **개선 방향**
  - 거래 체결 정책의 단일 진실 원천(SSOT)으로 백엔드 엔진을 지정한다.
  - 프론트는 주문 요청/유효성 UX만 담당하고, 실제 체결/정산 계산은 서버에서만 수행한다.

### 8-2. 공정성/보안 리스크 (클라이언트 직접 쓰기 권한)
- **현 상태**
  - Firestore Rules에서 사용자 본인 `users/{uid}`, `users/{uid}/portfolio` 쓰기가 허용되어 있다.
  - 클라이언트 트랜잭션이 직접 잔고/신용/포트폴리오를 갱신한다.
- **실제 위험**
  - 악의적 클라이언트가 비정상 payload를 구성할 경우, 서버 권위 없는 상태 변경 시도가 가능해진다.
  - 시즌 경쟁형 게임에서 “클라이언트가 상태를 확정”하는 구조는 공정성에 취약하다.
- **개선 방향**
  - 사용자는 주문 의도만 제출(`orders`), 체결 결과(`transactions`, `portfolio`, `balance`)는 서버만 갱신.
  - Rules를 “사용자 상태 직접 수정 금지 + 서버(Admin SDK/Function)만 쓰기” 구조로 재설계.

### 8-3. 정합성 이슈 (수수료/표시/계산식 불일치)
- **확인된 불일치 예시**
  - 거래 모달 표시 수수료(0.05%)와 실제 계산 수수료(0.1%)가 다를 수 있는 코드 경로가 존재한다.
  - 시즌3 전환 시 이런 UI-엔진 불일치가 신뢰도 문제로 바로 이어진다.
- **개선 방향**
  - 수수료율/이자율/정산 기준일을 상수화해 프론트·백엔드가 동일한 설정을 참조하도록 정리한다.
  - 화면 문구는 해당 상수를 렌더링해 “설명값”과 “실제 적용값”이 절대 어긋나지 않게 한다.

### 8-4. 데이터 모델 전환 리스크 (숏/USD 제거 이후)
- **현 상태**
  - 리더보드/대시보드/차트가 숏 포지션, 환율, 통화 분기를 전제로 계산하고 있다.
- **실제 위험**
  - 거래 로직만 바꾸고 집계식을 안 바꾸면 순자산/수익률/랭킹이 틀어진다.
- **개선 방향**
  - 시즌3 계산식 표준을 문서로 먼저 고정:
    - `Equity = Cash + LongMarketValue - UsedCredit`
    - KRW 단일 통화
    - 포트폴리오 수량 음수 금지
  - 리더보드/대시보드/미션/랭킹히스토리 집계식을 동일 표준으로 일괄 치환.

### 8-5. 운영 안정성 리스크 (배치/정산/장애 대응)
- **현 상태**
  - 스케줄러가 시세, 지정가, 미션, 검색, AI, 히스토리, 랭킹 기록을 한 프로세스에서 수행한다.
- **실제 위험**
  - 특정 기능 장애가 다른 핵심 배치(정산/체결)에 영향 줄 가능성.
  - 정산 배치 재실행/중복 실행 시 중복 처리 방지 장치가 약하면 금전 데이터 오류로 이어질 수 있다.
- **개선 방향**
  - 정산/체결 계열 배치에 idempotency key(중복 처리 방지 키) 도입.
  - 핵심 배치(정산/체결)와 부가 배치(AI/검색/히스토리) 실패 도메인을 분리.
  - 거래/정산 이벤트 감사 로그를 남겨 사후 재현 가능성 확보.

### 8-6. 시즌3 릴리즈 전 검증 항목 (추가)
- 거래경로별 결과 동일성: 시장가/지정가/재시도 모두 동일 결과인지 검증.
- 정산 시뮬레이션: T+2 영업일, 주말/공휴일, 잔고부족, 자동반대매매 케이스 테스트.
- 악성 시나리오 테스트: 클라이언트 직접 상태변경 시도 차단 검증.
- 계산 리그레션: 랭킹/대시보드/미션 값이 동일 기준식으로 산출되는지 샘플 계정 대조.

## 9) 운영 효율화 및 RTDB 최적화 (시즌 3 필수)

### 9-1. 장중 집중 운영 (Market Hours Lock)
- **운영 시간**: 한국 증시 시간(09:00 ~ 15:30)에만 초 단위/분 단위 고속 폴링 수행.
- **장외 시간**: 1시간에 1회 정도로 폴링 주기 대폭 완화. 
- **효과**: 불필요한 서버 자원 및 Firebase 연결/다운로드 비용 절감.

### 9-2. 스마트 업데이트 (Active vs Passive)
- **대상**: 전체 약 3,000개 종목(KOSPI, KOSDAQ, ETF) 중 선별 업데이트.
- **Active 갱신 (10~30초)**: '사용자 보유 종목' + '최근 검색 종목' + '상위 100위'.
- **Passive 갱신 (1~5분)**: 나머지 종목들.
- **효과**: RTDB 무료 요금제의 **다운로드(Egress) 한도(10GB/월)** 내에서 최대한의 실시간성 확보.

### 9-3. RTDB 샤딩 (Multi-Project 전략)
- **개념**: 무료 플랜의 한계(접속자 100명, 다운로드 10GB)를 극복하기 위해 데이터를 여러 프로젝트로 분산.
- **구조**:
    - **메인 프로젝트**: 인증(Auth), 유저 자산(Firestore), 거래 요청.
    - **시세 프로젝트 A**: KOSPI/ETF 실시간 가격 전용.
    - **시세 프로젝트 B**: KOSDAQ 실시간 가격 전용.
- **구현**: 프론트엔드에서 `initializeApp`을 여러 번 실행하여 각각의 시세 프로젝트 서버에 연결.

### 9-4. RTDB 무료 플랜 최적화 가이드라인
- **데이터 구조**: `price`와 `change` 등 필수 필드만 업데이트하여 패킷 크기 최소화.
- **구독 최적화**: 화면에 보이는 종목(약 10~20개)만 `onValue`로 실시간 구독하고, 나머지는 필요할 때만 단발성(`get`)으로 조회.
- **동시 접속자 관리**: 장시간 미활동 유저의 Firebase 연결을 클라이언트 단에서 자동으로 끊는 로직 추가.

### 9-5. 백엔드 마이크로 데몬 구조 (Service Separation)
- **개념**: 하나의 거대한 스케줄러 대신 역할별로 독립된 프로세스(데몬) 운영.
- **분리 세부**:
    - **Price Fetcher Daemon**: 시세 수집 및 시세 전용 프로젝트 A/B 업데이트 전담.
    - **Trade Engine Daemon (Server-side)**: 
        - 프론트엔드의 직접 쓰기 권한을 박탈하고, 유저는 주문 의도(`orders`)만 제출.
        - 백엔드에서 트랜잭션 처리 후 계좌/포트폴리오 확정. (보안 및 공정성 강화)
    - **AI & Logic Daemon**: AI 봇 루프, 미션 정산, 랭킹 집계 등 무거운 연산을 분리.
- **효과**: 특정 기능(예: AI API 지연) 장애가 핵심 시세/매매 시스템으로 전이되는 것을 차단.
